# 录波解析功能实现原理

## 一、功能概述

录波解析系统用于自动处理电力系统故障录波数据（COMTRADE 格式），支持两种处理路径：

- **简单解析**：上传单个 `.cfg` + `.dat` + `.hdr` 文件，生成基础分析报告
- **跳闸简报**：上传 ZIP 压缩包（含多个厂站、多套保护装置的录波文件），通过 LLM 自动生成结构化跳闸简报

---

```
前端 (React + Vite)

  ├── 上传文件（普通上传 / 分片上传）

  │

  ├── 跳闸简报管线  ──  webui/trip_briefing/
  └── LLM 调用      ──  webui/trip_briefing/llm/
```

## 三、数据流：从上传到出报告

### 3.1 文件上传

系统提供两种上传方式：

**普通上传** `POST /api/wave-record-parser/jobs`
- 直接 multipart 上传，适合小文件
- 参数：`files`（多个文件）、`station`、`device`、`device_type`

**分片上传**（适合大 ZIP 包，三步协议）：
1. `POST /uploads/init` — 初始化上传会话，传入文件名、总大小、分片数（每片 1MB）
2. `POST /uploads/{upload_id}/chunks/{index}` — 逐片上传
3. `POST /uploads/{upload_id}/complete` — 合并分片，创建任务

### 3.2 任务生命周期

任务在 SQLite `jobs` 表中流转，状态机如下：

```
queued ──→ processing ──→ completed
                │
                └──→ failed
```

- **队列处理**：使用 `asyncio.Lock` 保证同时只有一个任务在处理
- **进度回调**：处理过程中通过 `progress_callback` 更新 `progress`（0-100）和 `progress_message`
- **崩溃恢复**：服务重启时，所有 `processing` 状态的任务被标记为 `failed`

### 3.3 两条处理路径

任务创建后，`execute_job` 根据文件类型分流：

```
输入文件
  │
  ├── 有 .zip → 跳闸简报管线 (execute_trip_briefing)
  │
  └── 无 .zip，有 .cfg → 简单解析
        ├── parse_cfg_file()
        ├── parse_hdr_file()  [可选]
        └── generate_analysis_report() → analysis_report.md
```

## 四、COMTRADE 文件解析

### 4.1 CFG 文件（配置文件）

COMTRADE `.cfg` 是文本格式，定义了录波通道配置。解析逻辑：

```
第1行：厂站名, 装置名, 版本号
第2行：总通道数, 模拟通道数A, 数字通道数D
第3~N行：模拟通道定义（序号、名称、相别、单位、比例系数 a、偏移量 b）
后续行：数字通道定义、采样率、总采样数、起始时间、触发时间
```

**编码处理**：使用 `gbk`（简单解析）或 `gb18030`（跳闸简报管线）编码读取，兼容国内保护装置的中文输出。

系统中有两套 CFG 解析实现：
- `service.py:parse_cfg_file()` — 简单解析路径，提取基础信息
- `trip_briefing/scripts/parse_dat_to_csv.py:parse_cfg()` — 跳闸简报路径，返回完整的 `Cfg` 数据类，含 CT/PT 变比、采样率等

### 4.2 DAT 文件（数据文件）

DAT 文件存储实际的采样数据，支持 ASCII 和 Binary 两种格式：

**ASCII 格式**：每行 `采样序号, 时间戳(μs), A1, A2, ..., D1, D2, ...`
- 模拟量经过线性变换：`实际值 = 原始值 × a + b`

**Binary 格式**：
- 4 字节采样序号 + 4 字节时间戳 + N×2 字节模拟量（int16）+ M×2 字节数字量（uint16，每位代表一个数字通道）
- 同样应用线性变换

**高性能解析**：`parser.py:parse_dat_fast()` 使用 `numpy.frombuffer` 一次性读取整个二进制文件，向量化计算缩放，比逐行解析快 10-50 倍。

### 4.3 HDR 文件（头信息文件）

HDR 是 XML 格式，包含装置的详细元数据：

```xml
<HDR>
  <DeviceInfo>      ← 装置基本信息（名称/值对）
  <FaultInfo>       ← 故障信息（名称/值/单位）
  <TripInfo>        ← 跳闸信息（子标签/值对）
  <DigitalEvent>    ← 数字事件记录
  <CTPTAnalysis>    ← CT/PT 变比分析
</HDR>
```

## 五、RMS 计算与事件提取

`calculate_rms.py:extract_statistics_and_events()` 是数据处理的核心，约 500 行代码。

### 5.1 RMS（有效值）计算

```
采样数据 → 滑动窗口 RMS → 半周波差分 → 突变检测
```

1. **计算每周期采样点数**：`samples_per_cycle = 采样率 / 频率(50Hz)`
2. **滑动窗口 RMS**：以 `samples_per_cycle` 为窗口大小，居中计算均方根值
3. **半周波差分**：`diff[i] = rms[i + half_cycle] - rms[i]`
   - 正向突变（diff > 0）表示故障发生
   - 负向突变（diff < 0）表示故障切除

### 5.2 突变检测算法

对每个模拟通道，检测四类突变：

| 突变类型 | 判据 | 含义 |
|---------|------|------|
| 第一次正突变 | diff > RMS最大值 × 20% | 故障发生 |
| 第一次负突变 | diff < -RMS最大值 × 20% | 故障切除 |
| 第二次正突变 | 故障切除后电流恢复，再次突增 | 重合闸后再次故障 |
| 第二次负突变 | 再次故障后再次切除 | 重合闸失败 |

**参考相识别**：保护电流通道中正突变最大的一相作为参考电流相。

### 5.3 数字事件提取

数字通道的状态变化被记录为事件：
- `0 → 1`：动作（如保护启动、出口跳闸）
- `1 → 0`：返回（如保护复归）

### 5.4 输出文件

每个装置目录下生成三个文件：
- `.rms.csv` — RMS 统计表（行为统计指标，列为通道）
- `.events.csv` — 数字事件表（按绝对时间排序）
- `.current_mutation.csv` — 电流突变汇总（A/B/C 相正负突变）

## 六、跳闸简报管线（Trip Briefing Pipeline）

### 6.1 管线总览

`pipeline.py:run_pipeline()` 编排整个流程：

```
Step 0  准备工作目录（解压 ZIP，修复编码）     进度 30%
Step 1  COMTRADE 数据处理（DAT→CSV，RMS计算）  进度 35%
Step 2  扫描装置目录                           进度 40%
Step 3  创建 LLM 客户端                        进度 45%
Step 4  逐装置生成段落（Subagent LLM）          进度 45%-85%
Step 5  生成跳闸简报（Main Agent LLM）          进度 90%
Step 6  保存监控日志                            进度 93%
```

### 6.2 ZIP 解压与编码修复

`_fix_zip_encoding()` 处理 Windows 系统打包的 ZIP 文件中文目录名乱码问题：
- 检测文件名编码（尝试 GBK → UTF-8 → 原样）
- 重新写入正确编码的文件名

### 6.3 装置扫描

`scan_devices()` 递归扫描两类目录结构：

```
保护录波/                          故障录波/
├── 厂站A/                         ├── 厂站A/
│   ├── 第一套保护/                 │   ├── *.cfg, *.dat, *.hdr
│   │   ├── *.cfg, *.dat, *.hdr    │   └── *.rms.csv, *.events.csv
│   │   └── *.rms.csv, *.events.csv│
│   └── 第二套保护/                 └── 厂站B/
│       └── ...                         └── ...
```

每个装置被建模为 `DeviceFiles` 对象，包含厂站名、套别、以及 `.hdr`、`.rms.csv`、`.events.csv` 文件路径。

### 6.4 LLM 调用：两级架构

系统采用 **Subagent + Main Agent** 两级 LLM 调用：

```
┌─────────────────────────────────────────────┐
│  每个装置 → Subagent LLM → 一段分析文字     │
│  (保护装置A_第一套保护 → 段落1.md)           │
│  (保护装置B_第二套保护 → 段落2.md)           │
│  (故障录波器A         → 段落3.md)           │
└─────────────────────────────────────────────┘
                    │
                    ▼ 所有段落拼接
┌─────────────────────────────────────────────┐
│  Main Agent LLM → 跳闸简报.md               │
└─────────────────────────────────────────────┘
```

## 七、LLM Prompt 设计

### 7.1 Subagent Prompt（装置级分析）

每个设备类型的 Subagent Prompt 接收三份原始数据：
- **HDR 内容**：装置元数据、故障信息、跳闸信息
- **RMS CSV**：各通道有效值统计
- **Events CSV**：数字事件表

输出要求为结构化 Markdown，包含：
- 装置信息、HDR 信息（DeviceInfo、TripInfo、DigitalEvent、FaultInfo）
- 事件表、RMS 统计（电流最大值、电压最小值、一次/二次突变）
- **关键规则**：只使用绝对时间，CT/PT 变比从 HDR SettingValue 取值

**按设备类型差异化**：

| 设备类型 | 子代理 Prompt 特点 |
|---------|-------------------|
| 线路 (line) | 故障测距、线路长度、重合闸分析；CT换算：一次侧(kA) = 二次侧(A) × CT一次 / CT二次 / 1000 |
| 主变 (transformer) | 高/中/低压侧分析、差动电流、变压器类型识别（T2=220kV双低压分支，T5=500kV无中压） |
| 母差 (bus) | 大差/小差、间隔级电流分析、间隔-母线映射、复合电压闭锁 |
| 故障录波 | 简化版，无 CT/PT 设置节，数据汇入上层设备的 Main Agent |

### 7.2 Main Agent Prompt（简报级综合）

Main Agent 接收所有装置段落，生成最终跳闸简报。以线路为例：

1. **故障概况表**：时间、厂站、装置、相别、故障类型、保护动作、故障电流、故障距离、重合闸
2. **保护配置**：装置信息表、通道配置（含 CT/PT 变比）
3. **各套保护动作详情**
4. **动作时序表**：时间对齐（电流正突变对齐 → 保护启动对齐）
5. **综合分析**：双套一致性、差动电流、故障距离校验、重合闸分析
6. **故障判断结论**
7. **故障录波器评价**

**交叉验证规则**（强制执行）：
- 故障距离：本侧 + 对侧 ≈ 线路全长（误差 < 20%）
- 差动电流：双侧偏差 < 10%
- 动作时序：同厂站双套时间差 < 20ms
- 电流一致性：保护 vs 故障录波同相偏差 < 20%
- 数据矛盾必须标注，不得强行统一

## 八、LLM 客户端

`trip_briefing/llm/client.py` 封装了 OpenAI 兼容 API 调用：

- 默认模型：`qwen3.5-flash`，temperature=0.1
- 支持 `enable_thinking` 模式（通过 `extra_body` 传递）
- 重试策略：最多 3 次，指数退避（2^attempt 秒）
- 超时：180 秒/次调用

LLM 配置从 `~/.protection/config.json` 或 `~/.nanobot/config.json` 读取，按优先级选择 provider：zhipu > dashscope > deepseek > openai > openrouter。

## 九、监控与日志

`trip_briefing/monitor.py` 的 `Monitor` 类跟踪每个步骤：
- 墙钟耗时
- Token 用量（输入/输出/总计）
- 成功/失败状态

最终输出 `token_usage.log` JSON 文件，含逐步详情和汇总（总耗时、总 Token、处理装置数）。
