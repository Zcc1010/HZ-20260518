# 主 Agent 跳闸简报生成 Prompt 模板

## 任务

**只读段落文件**，基于所有 Subagent 生成的装置段落（Markdown 格式），生成完整的跳闸简报。

## 前置检查门（生成简报前强制执行 · 缺一项即停止）

**在读取任何段落文件之前，必须逐项检查以下产物是否存在。若任一项缺失，唯一允许的响应是报告缺失项并停止，不得以任何理由绕行。**

| 检查项 | 产物 | 检查方式 | 若缺失 |
|--------|------|----------|--------|
| 步骤 0 | `parse_folder_name.py --json` 输出的厂站/电压/套别/设备类型映射 | 查找 JSON 或 确认已执行 | **停止，报告"步骤0未执行"** |
| 步骤 5 | 每台装置一份段落文件（`段落/{厂站}_{设备}.md`） | `ls` 段落目录，对照步骤 0 的装置清单 | **停止，报告"步骤5未完成，缺以下装置段落：{列表}"** |
| 段落完整性 | 每份段落含三区块（HDR信息 / Events信息 / RMS信息）+ 前置条件 | 读取每份段落，检查是否含所有区块标题 | **停止，报告"段落不完整：{文件名} 缺 {区块名}"** |

> **此门的目的**：让"跳过步骤"在物理上不可能——如果没有段落文件，主 Agent 就没有数据可以读，只能报告流程断裂。绵动线案例中主 Agent 之所以能绕过，正是因为"自己读 HDR"比"等段落生成"更快——这道门切断了那条路径。

**执行方式**：主 Agent 生成简报前，必须先执行：
```bash
python <skill>/scripts/check_pipeline.py <分析目录>
```
若返回非零退出码，**立即停止**，报告缺失的步骤，不进行任何简报生成。

## 流程红线（违例即无效 · 绝对优先）

**以下三条为主 Agent 不可逾越的边界：**

1. **主 Agent 仅读段落文件**：禁止 `Read` 任何原始 HDR/events/RMS/CFG/DAT 文件。段落是数据层与报告层的写保护边界。
2. **不越俎代庖**：Subagent 段落缺失/不完整时，主 Agent 的职责是**要求重新生成该段落**，而非自行从原始文件提取数据补全。
3. **数据溯源必须标注段落来源**：简报中的每项数据标注"来自 {段落文件名}"，段落自身应在相应字段注明"来源：{HDR/events/RMS 文件名}"。

> **Why**（绵动线教训）：主 Agent 因"数据看起来完整"跳过步骤 0（parse_folder_name.py）和步骤 5（Subagent 段落），直接读取原始 HDR/events/RMS 生成简报——导致厂站识别错误、报告结构失控、三次分析三种结果。流程即是质量，跳步即是隐患。

**装置参数必提取清单（生成简报前必须完成）**：
每台保护装置须从 HDR SettingValue 提取并填入报告：CT一次/二次额定值、PT一次额定值、线路总长度、差动动作电流定值、接地距离Ⅰ段定值、线路正序/零序阻抗。NSR 装置另查 CFG 第1行+模拟通道比例。**禁止在 CT/PT 为空的情况下出具结论或一次值换算**。

**报告布局强制规则（必须遵守）**：
- **PDF 非默认**：输出仅含 `.md` 简报；**除非用户明确要求，否则不得自动生成 PDF**。
- **动作时序表按厂站分开**：双端案例禁止合并成融合表，必须拆为 §N.1 + §N.2 两张独立表（见 §4.2）。**同厂站多套保护分列展示**：列为「相对时刻 / 第1套 / 第2套」，禁止用行标注"装置"列区分。）
- **保护动作时序表仅列保护装置**：故障录波器为监测设备，**不得**出现在时序表中；录波器数据仅用于 §5.4 故障录波评价交叉验证。
- **保护与录波器分表**：§二"涉及保护装置"中保护装置与故障录波器为两张独立表，禁止混入；保护表"厂站名称"列为第二列。
- **电气量互校表默认不列录波器**：表中仅列保护装置。录波器仅在**同厂站两套保护电气量不一致**（偏差≥10%）时作为仲裁参考引入，注明原因。
- **两侧差流一次值互校（双端线路必做）**：差动保护动作的线路故障，取各装置 B 相差流**最大值**（WXH-803A HDR 为动作瞬间值不可用，必须从 DAT RMS 提取）。两侧一次值偏差 < 10% 为一致，≥ 10% 标 `[待核实]`。
- **时序分析 > HDR + DAT 综合 + 正突变统一对齐**：时序表必须综合 HDR（动作）和 DAT（返回/位置）。所有装置通过故障相电流正突变对齐到同一时间轴（正突变=同一物理事件的表现），以最早校正保护启动为 t=0ms。**严禁各装置以自身保护启动为独立 t=0 直接比较**。HDR/DAT 差≤3ms 合并，>3ms 标注。
- **§一 故障线路行仅含线路参数**：格式「线路名（厂站1↔厂站2），电压等级，全长 XX km；正序阻抗 XXΩ、零序阻抗 XXΩ」。**不含** CT/PT 和Ⅰ段定值。
- **§一 故障位置含两侧距离及占比**：格式「距厂站1约 XX km（占全长约 XX%），距厂站2约 XX km（占全长约 XX%）」。
- **表格简洁**：列头用「时刻」不用「相对时刻」；装置列用型号+简称；编号用「#」不用「序号」。

## 新段落格式（HDR/Events/RMS）

段落按三个区块组织：

### HDR信息区块
包含：
- **设备信息**：装置型号、程序版本等
- **TripInfo（保护动作时间序列）**：表格，含绝对时间、动作名称、相别、值
- **DigitalEvent（数字量事件）**：表格，含绝对时间、名称、变化
- **定值信息**：差动定值、线路长度、重合闸定值等关键定值

### Events信息区块
表格，含：
- 绝对时间
- 通道名称（原始名称）
- 内容（动作/返回）

### RMS信息区块
包含：
- 电流电压RMS最大值
- 电压最小值
- 电流电压突变信息

## 生成规则

### 数据来源约束

- **只读段落文件**，禁止读取原始 hdr/rms/events/csv/dat 文件
- 段落中已有的数据，直接使用，不修改
- 发现段落数据矛盾时，在对应位置标注疑问，不覆盖
- **厂站 / 设备名称 / 保护套别 / 电压等级 / 装置型号：优先采用 `parse_folder_name.py` 从目录名推断的结果**（已在各段落"前置条件"落地）；HDR 与目录名冲突时以 HDR 为准，并在对应位置标注 `[目录名推断与 HDR 不一致：…]`。目录名无法确定的项（如断路器目录常无电压）按段落 `[待核实]` 注记处理。
- 段落未覆盖的数据（如线路名称），从目录结构或文件名推断
- **每条数据必须能溯源到特定段落**：在综合分析等章节引用数据时，注明来源（如"崇本变第1套段落显示..."）

### 段落数据读取方法

从新格式段落中读取数据：

| 跳闸简报数据项 | 段落来源 |
|----------------|----------|
| 装置型号/版本 | HDR信息 → 设备信息 |
| 制造厂家 | 段落开头字段 |
| 保护动作时间 | HDR信息 → TripInfo + Events信息 |
| 故障电流 | RMS信息 + HDR信息 → 定值信息(CT变比) |
| 差动电流 | RMS信息 + HDR信息 → 定值信息(差动定值) |
| 测距 | HDR信息 → TripInfo/定值信息 |
| 重合闸方式 | HDR信息 → 定值信息 |
| CT变比 | HDR信息 → 定值信息 或 定值与变比信息区块（标注[无数据]） |
| PT变比 | HDR信息 → 定值信息（标注[无数据]） |
| 线路全长 | HDR信息 → 定值信息 |
| 差动动作电流定值 | HDR信息 → 定值信息 或 定值与变比信息区块 |
| 距离保护Ⅰ段有效(控制字) | HDR信息 → 定值信息 "距离保护Ⅰ段有效" |
| 距离I段保护范围 | 段落 "距离I段不动作原因" 区块 |
| 差流一次值校验 | RMS信息 差流二次值 + 定值信息 CT变比 |

**距离I段不动作触发读取**：
- 当4套Events均无距离I段动作 → 必须从各套段落读取"距离I段不动作原因"区块
- 输出至主简报 §5.5 距离I段不动作原因判别（详见下文）

### 一致性校验

对段落中的数据进行交叉校验，对超过以下阈值的数据标注 `[待核实]`：

1. **测距校验**：本侧测距 + 对侧测距 ≈ 线路全长（允许误差 20%），超出标注 `[待核实: 测距偏差过大]`
2. **差流一次值校验**：两侧四套差流一次值偏差 > 10% 时标注 `[待核实: 差流偏差较大]`（必填）
3. **动作时间校验**：同厂站两套保护动作时间差 > 20ms 时标注 `[待核实: 动作时间差较大]`
4. **电流一致性**：故障录波电流与保护录波同相电流偏差 > 20% 时标注 `[待核实: 电流一致性存疑]`
5. **距离I段判别**：4套均无距离I段动作 且 控制字=1 且 故障点 ≤ I段范围 → `[待核实: 距离I段应动未动]`

标注 `[待核实]` 的数据不得删除，但也不得自行修正，在最终结论中说明待核实项。

### 段落缺失处理

- 若某套段落缺失，在对应位置标注`[第X套数据缺失]`
- 简报其他部分基于已有数据生成

### 故障类型判断

- 重合成功 → 瞬时性故障
- 停用重合闸/重合失败/未重合 → 永久故障
