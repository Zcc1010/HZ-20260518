# 检修时段过滤规则

> 运行风险评估智能体必须结合现场检修工作做告警过滤——
> 检修时段内的告警**几乎都不是装置本体风险**，
> 而是检修操作的预期现象（如：装置消缺重启中瞬时通讯中断、
> 采样板卡更换期间参数异常等）。

## 一、为什么必须过滤

**反例：** 不做检修过滤，则会出现这种"假危急"：

1. 某装置昨日 10:00-12:00 报"通道一无有效帧"（通道光电模块更换期间）
2. 评估引擎判定"未复归告警，闭合主保护存在风险 → 危急"
3. 现场实际：通道已修复，仅缺一次定值区切换

正确做法：
- 该告警 timestamp 落在检修窗口 → **过滤**
- 评估结论降级
- 但保留"该装置近期有检修"的提示，给评估报告做上下文

### 调用

```bash
python scripts/run_risk_assessment.py \
    --maintenance-file 保护装置信息/保护装置检修工作.json
```

### 数据缺失时

`load_maintenance` 在文件不存在时 noop，
但不报错——所有评估按"无可用检修过滤"运行。

## 三、过滤级别

| 级别 | 处理 |
|---|---|
| **L1 默认** | 检修时段内的告警**完全不算风险** |
| **L2 频发抖动** | 检修时段内的告警**也参与抖动统计**（保留上下文） |
| **L3 反向提示** | 复役后短时间内复发 → 升级 |

默认实现为 L1 + L3：
- L1: `alarm_in_maintenance_window()` 直接返回 True 则忽略该告警
- L3: 检修窗口外的频发抖动仍计入；这一点的判别在 `analyze_alarm_state`

## 四、故障案例：检修窗口不对齐

**场景：**
- 调度登记检修时间：08:00-18:00
- 实际检修操作结束时间：16:30
- 但 17:00 出现真实告警（后续生产异常）
- 程序会过滤 17:00 的告警 → 漏报

**对策：**
- 检修结束时间应留 buffer（如登记 18:00，实际 16:30 结束）
- 程序不自动加 buffer，等待人工修订

## 五、数据治理建议

| 来源 | 维护方式 |
|---|---|
| 调度 OMS / 工作票系统 | 抽取，按日同步到 `保护装置检修工作.json` |
| 现场纸质记录 | 班组建模录入 |
| 反措消缺闭环 | 与缺陷管理流程对接 |
